# Cross-Review Skill

複数のドキュメントやコードファイルを横断的にレビューし、整合性・正確性・網羅性を検証します。

## 使用方法

```
/cross-review <ファイルパス1> <ファイルパス2> [ファイルパス3 ...] [--output <出力パス>]
```

### 引数

- **ファイルパス**（必須、2つ以上）: レビュー対象のファイルパス（スペース区切り）
- **--output**: レビューレポートの出力先（省略時: 標準出力）

## 処理手順

### Step 1: 引数解析

$ARGUMENTS を解析し、ファイルパスとオプションを分離する。

- `--output` の次の引数を出力先パスとして取得
- それ以外をレビュー対象ファイルパスとして収集
- Glob パターン（`*.md` 等）が含まれる場合は展開する

ファイルが2つ未満の場合はエラーメッセージを表示して終了:
```
使い方: /cross-review <ファイル1> <ファイル2> [ファイル3 ...] [--output <出力パス>]
例: /cross-review docs/kachaka_spec.md docs/temi_spec.md
```

### Step 2: 対象ファイル読み込み

すべての対象ファイルを Read ツールで読み込む。
- ファイルが存在しない場合はエラーとして報告
- 大きなファイル（2000行超）は構造把握のため先頭と末尾を中心に読み込む

各ファイルについて以下を把握する:
- ファイルの種別（仕様書、設計書、コード、設定ファイル等）
- 全体構造（セクション構成、章立て）
- 主要な定義・データ型・インターフェース

### Step 3: 5観点レビュー

以下の5つの観点で各ファイルを横断的にレビューする。
必ずすべての観点を実施し、それぞれについて具体的な指摘を行うこと。

#### 観点1: 正確性（Accuracy）

技術的な誤りや矛盾がないかチェックする。

- データ型の不一致（String vs Int、required vs optional 等）
- API仕様とコードの乖離
- 数値・定数・Enum値の誤り
- プロトコル仕様との不一致
- コード例やサンプルの動作可否

#### 観点2: 網羅性（Completeness）

抜け漏れがないかチェックする。

- あるドキュメントで定義されているがほかで触れられていない項目
- エラーケース・異常系の考慮漏れ
- 必要なフィールド・パラメータの定義漏れ
- テストケースや検証項目の不足
- TODOや未実装マークの残存

#### 観点3: 整合性（Consistency）

ドキュメント間で一貫しているかチェックする。

- 用語・命名の揺れ（例: robotId vs robot_id vs thingId）
- 同一概念に対する異なる説明・定義
- JSON構造・フィールド名の不一致
- バージョン番号・日付の不一致
- フォーマット・記法の統一性（snake_case vs camelCase 等）

#### 観点4: 実用性（Usability）

実装者が参照して迷わないかチェックする。

- 曖昧な記述（「適切に」「必要に応じて」等の具体性のない表現）
- 実装手順の欠落や不明確さ
- 設定値の推奨値・デフォルト値の不明
- サンプルコードの有無と品質
- 読み手の前提知識の過不足

#### 観点5: リスク（Risk）

見落とされているリスクや課題をチェックする。

- セキュリティ上の懸念（認証・認可、入力バリデーション）
- パフォーマンスのボトルネック
- 障害時の復旧手順の欠如
- 外部依存による影響範囲
- スケーラビリティの制約
- 後方互換性の問題

### Step 4: レビューレポート生成

以下のテンプレートに従ってレビューレポートを生成する。

### Step 5: 出力

- `--output` が指定されている場合は Write ツールでファイルに書き出す
- 未指定の場合はそのまま出力する（ファイル作成しない）

## レビューレポートテンプレート

```markdown
# Cross-Review Report

| 項目 | 値 |
|------|-----|
| レビュー日 | YYYY-MM-DD |
| 対象ファイル数 | N |

## 対象ファイル

| # | ファイル | 種別 | 行数 |
|---|---------|------|------|
| 1 | path/to/file1.md | 仕様書 | 350 |
| 2 | path/to/file2.md | 仕様書 | 420 |

---

## 重大な指摘（修正必須）

指摘がない場合は「なし」と記載。

### [CR-001] 指摘タイトル
- **観点**: 正確性 / 網羅性 / 整合性 / 実用性 / リスク
- **対象**: ファイル名 > セクション名（行番号があれば記載）
- **内容**: 具体的な問題の説明
- **修正案**: 具体的な修正内容

### [CR-002] ...

---

## 軽微な指摘（修正推奨）

### [CR-0XX] 指摘タイトル
- **観点**: ...
- **対象**: ...
- **内容**: ...
- **修正案**: ...

---

## 改善提案（あれば良い）

### [CR-0XX] 提案タイトル
- **観点**: ...
- **対象**: ...
- **内容**: ...

---

## 良い点（特筆すべき点）

- **[ファイル名]**: 良い点の説明
- **[ファイル名]**: 良い点の説明

---

## 総合評価

### 観点別スコア

| 観点 | スコア | コメント |
|------|--------|----------|
| 正確性 | A/B/C/D | 一言コメント |
| 網羅性 | A/B/C/D | 一言コメント |
| 整合性 | A/B/C/D | 一言コメント |
| 実用性 | A/B/C/D | 一言コメント |
| リスク | A/B/C/D | 一言コメント |

### スコア基準
- **A**: 問題なし、または軽微な指摘のみ
- **B**: いくつかの修正推奨事項あり
- **C**: 重大な指摘あり、修正が必要
- **D**: 複数の重大な指摘あり、大幅な見直しが必要

### サマリー

レビュー全体を通じた総括コメント（3-5行）。
特に優先して対応すべき点と、全体の品質に対する評価を記載。

### 指摘件数サマリー

| 区分 | 件数 |
|------|------|
| 重大な指摘 | X 件 |
| 軽微な指摘 | X 件 |
| 改善提案 | X 件 |
| 良い点 | X 件 |
```

## 注意事項

- 指摘には必ず具体的なファイル名・セクション名・行番号を含めること
- 推測や曖昧な指摘は避け、根拠を明示すること
- 指摘番号は CR-001 から通し番号で振ること
- 修正案は実行可能な具体的な内容にすること
- 良い点も必ず記載し、バランスの取れたレビューにすること
- 出力は日本語で記述する
